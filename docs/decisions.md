## D-001 マルチDB構成(thinkmotion_db / client_db / code_db)

- **決定**: アプリケーションごとにDBを分離する(`thinkmotion_db`をデフォルト、`client_db`をPilates用、`code_db`をCode用として確保)
- **背景**: 「構造化セルフケアプラットフォーム」として複数アプリケーション(Pilates / ThinkMotion / Code / Admin)を持つ構成のため、データの責務を明確に分けたい
- **理由**: 各アプリケーションの成長速度・スキーマ変更頻度が異なることが見込まれるため、DBレベルで分離しておくと将来の切り出し(マイクロサービス化やアプリごとの独立運用)がしやすい
- **トレードオフ**: JOINを跨いだ集計ができない。`users`テーブルの共有方法(D-002)で一部緩和

## D-002 usersテーブルの共有

- **決定**: Pilates / ThinkMotionで登録入口(フォーム・フロー)は分けるが、`users`テーブルは1つに統合する
- **背景**: 同一人物が理学療法士としてThinkMotionを使い、同時にクライアントとしてPilatesを予約するケースが想定される
- **理由**: アカウントを分けると同一人物の名寄せができず、将来的な横断機能(例: 会員向け特典の統合)が困難になる
- **トレードオフ**: マルチDB構成(D-001)との整合性のため、どちらのDBに`users`テーブルを置くか、または参照方法を明確にする必要がある(要確認: 現状の実装がどちらに寄せているか)

## D-003 is_paid_venue

- **決定**: 会場ごとに追加料金が発生するかどうかをbooleanの`is_paid_venue`で表現する。金額は会場マスタにデフォルト値(`surcharge`)として持たせつつ、予約単位での上書きも許容する
- **背景**: beautyRubyの¥1,000/人は交通費・場所代を区別せず一律の追加料金であり、公民館は基本追加料金が発生しない(使用料が高い会場では将来的にお客様に負担してもらう可能性はゼロではない)。ただし公民館では予約する部屋を変更する必要がある場合など、同じ会場でも金額が変わる正当なケースがある
- **理由**: 「有料/無料」の二値かつ会場デフォルト値で大半のケースはカバーできるが、部屋変更のような会場側の事情による金額変動には対応できない。フリー入力の上書きを予約単位で許容することで、例外を無理に会場マスタ側の複雑化(部屋ごとのマスタ化等)で吸収せず、実際に発生した金額をそのまま記録できる。青色申告上「交通費」と「場所代」を経費区分として分けて記帳する必要はあるが、これは会計処理側の分類の問題であり、予約システムのDB設計上は単一の`surcharge`で表現して問題ない
- **トレードオフ**: 上書きを自由記述にする分、「なぜこの予約だけ金額が違うのか」を後から追いにくくなる可能性がある。乱用を防ぐ運用ルール(誰が・いつ・なぜ変更したかの記録)は別途検討の余地あり

## D-004 guest_pass_purchases + guest_pass_usages への分離

- **決定**: 回数券を「購入履歴」と「利用履歴」の2テーブルに分離し、残数は都度計算する
- **背景**: 当初は`guest_passes`という単一テーブルで残数をカラムとして直接持たせる設計だった
- **理由**: 残数をカラムで直接管理すると、予約キャンセルや不整合発生時に手動で数値を調整する必要が生じ、データの正当性を保証しづらい。購入と利用を分離し残数を動的計算にすることで、常に履歴から残数を再現できる(監査可能性の担保)
- **トレードオフ**: 残数取得のたびに集計クエリが必要になるため、パフォーマンスが問題になった場合はキャッシュ層の検討が必要

## D-005 キャンセルポリシーの一本化(¥500定額)

- **決定**: 会場や予約種別によらず、前日正午締め切り後のキャンセル料を¥500定額とする
- **背景**: 会場ごとに異なるキャンセルポリシーを設定する案もあった
- **理由**: 料金体系が複雑になるとクライアントへの説明コストが増し、運用ミスの原因にもなる。少人数運営の現状では、シンプルなルールの方が講師・クライアント双方にとって認知負荷が低い
- **トレードオフ**: 会場側の事情(公民館のキャンセル規定等)と自社ポリシーが将来乖離した場合は例外処理が必要になる可能性がある

---

## D-006 no_showステータスの自動ブロックなし

- **決定**: `no_show`ステータスを予約に持たせるが、一定回数での自動制限(次回予約ブロック等)は行わない
- **背景**: 無断キャンセル・当日不参加の記録機能自体は必要だが、自動化した罰則ルールにするかは別問題
- **理由**: 小規模運営では個々の事情(体調・家庭の事情等)が多様であり、機械的なルールよりも講師の目視判断を優先する方針(「小規模ゆえに人間の判断を残す」という設計思想と一致)
- **トレードオフ**: 記録が講師の確認頻度に依存するため、確認を怠ると意味をなさない。運用ルール(いつ確認するか)は別途明文化が必要

## D-007 CancelReservationアクションクラスへの集約

- **決定**: クライアント側・講師側それぞれのキャンセル処理を、`cancelledBy`(`client` / `teacher`)を引数に取る単一のアクションクラスに集約する
- **背景**: ユーザー側キャンセル機能を先に実装した際、キャンセル時の副作用(ステータス変更・`cancelled_at`記録・キャンセル料判定等)がコントローラに直接書かれていた
- **理由**: 管理側キャンセル機能を追加するにあたり、同じロジックを2箇所に重複させたくない。誰がキャンセルしたかによって分岐する部分(通知内容、料金判定の可否等)はあるが、コア処理は共通化できる
- **状態**: 🔧着手中。ユーザー側は実装済み、管理側実装時に抽出予定
- **トレードオフ**: 分岐ロジックが増えるとアクションクラス自体が肥大化するリスクがあるため、`client`/`teacher`以外の呼び出し元が増える場合は再分割を検討

## D-008 通知優先度(講師への通知 > 自動クライアント通知)

- **決定**: LINE Messaging APIによる通知のうち、講師自身へのプッシュ通知を自動クライアント通知より優先して実装する
- **背景**: 通知機能は複数の宛先(講師/クライアント)・複数のトリガー(予約確定/キャンセル/リマインド等)が考えられる
- **理由**: 小規模運営では、講師が予約状況をリアルタイムに把握できることの方が、クライアントへの自動通知よりも運営上の優先度が高い(見落としによる二重予約等の実害を防ぐ)
- **トレードオフ**: クライアント向け通知(リマインド等)の実装が後回しになる分、当面は手動対応やマイページでの自己確認に依存する

## D-009 ThinkMotionの非ステータス設計(いいね・フォロワー数なし)

- **決定**: いいね数・フォロワー数などのステータス指標を実装しない
- **背景**: 知識共有プラットフォームとしては、いいね数等によるエンゲージメント設計が一般的
- **理由**: 「キラキラ系ではない」というプロダクト全体のコンセプトに基づき、可視化された数値による承認欲求・比較を誘発する設計を意図的に避ける。目的は「思考を磨く場」であり、人気競争の場ではない
- **トレードオフ**: エンゲージメント指標がないため、利用状況の把握には別の指標(投稿頻度、閲覧数等)を検討する必要がある

---

## D-010 セッション一覧の月単位ナビゲーション

- **決定**: 管理画面のセッション一覧は、月単位のナビゲーションを主UIとする
- **背景**: セッション一覧の表示方法として、リスト全件表示や週単位表示等の選択肢もあった
- **理由**: 青色申告のCSV出力が月次処理であるため、UI上の区切りを月に合わせることで「その月の実績を確認してエクスポートする」という業務フローと画面構造を一致させる
- **トレードオフ**: 月をまたぐ分析(トレンド確認等)には別途集計画面(分析)が必要になる
