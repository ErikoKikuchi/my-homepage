# Requirements

本ドキュメントでは、本プロジェクトで実装する機能要件を管理します。
設計判断は decisions.md、DB設計は db-schema.md、API仕様は api.md に記載します。実装済み/計画中の状態を凡例で区別します。

凡例: ✅実装済み ／ 🔧着手中 ／ 📋未着手(仕様確定) ／ 💭検討中(仕様未確定)

---

## 0. 共通基盤

| 項目                 | 内容                                                                                     | 状態                |
| -------------------- | ---------------------------------------------------------------------------------------- | ------------------- |
| 技術スタック         | Laravel 12 / PHP 8.3 / MySQL 8.0 / Docker / Nginx / Tailwind CSS                         | ✅                  |
| DB構成               | `thinkmotion_db`(デフォルト) / `client_db`(Pilates) / `code_db`(Code、検討中) のマルチDB | ✅(code_dbは箱のみ) |
| ユーザー管理         | Pilates / ThinkMotionで登録入口を分けつつ、`users`テーブルは共有                         | ✅                  |
| 認証                 | Fortifyによるマルチガード(`web`, `admin`)                                                | ✅                  |
| 管理者認証           | 独自ガード + 2FA(pragmarx/google2fa-laravel)                                             | ✅                  |
| メール認証           | 会員登録時のメール認証フロー                                                             | ✅                  |
| ログインリダイレクト | `?from=`クエリパラメータ + カスタム`LoginResponse`                                       | ✅                  |
| ルーティング分離     | `web.php` / `thinkmotion.php` / `client.php`を`bootstrap/app.php`で登録                  | ✅                  |

## 1. Pilates(予約・自主トレログ / BodyMind)

### 1-1. 予約機能

| 項目               | 内容                                                                                                     | 状態 |
| ------------------ | -------------------------------------------------------------------------------------------------------- | ---- |
| カレンダー表示     | 空きあり / 要相談 / 満席の3状態                                                                          | ✅   |
| 空き枠ロジック     | `ReservationAvailabilityService`に集約                                                                   | ✅   |
| 予約リードタイム   | 前々日正午締め切り(config管理)                                                                           | ✅   |
| 会場               | 自スタジオ / 公民館 / beautyRuby(苫小牧、外部提携)                                                       | ✅   |
| 会場ごとの追加料金 | `is_paid_venue`で追加料金の有無を判定、金額は会場マスタにデフォルト値(surcharge)、予約単位での上書きも可 | 📋   |
| beautyRuby追加料金 | ¥1,000/人(交通費・場所代を区別しない一律料金)、`is_bookable`で顧客向け表示を制御                         | 📋   |

### 1-2. 回数券・決済

| 項目       | 内容                                                                    | 状態 |
| ---------- | ----------------------------------------------------------------------- | ---- |
| 回数券管理 | `guest_pass_purchases` + `guest_pass_usages`の2テーブル、残数は動的計算 | 📋   |
| 決済手段   | Stripe / 現金                                                           | 📋   |
| 会計連携   | `memo`欄で青色申告用CSV出力に対応                                       | 📋   |

### 1-3. キャンセル

| 項目                      | 内容                                                            | 状態 |
| ------------------------- | --------------------------------------------------------------- | ---- |
| キャンセルポリシー        | 前日正午締め切り後は¥500定額                                    | 📋   |
| キャンセル記録            | `cancelled_at` / `cancelled_by`で記録                           | ✅   |
| ユーザー側キャンセル導線  | マイページからページ遷移方式(モーダルでない)                    | ✅   |
| 未収金ブロック            | `has_unpaid_fee`フラグで予約確定をブロック、手動解除            | 📋   |
| no_showステータス         | 講師の目視判断による記録、自動ブロックはしない                  | 📋   |
| 管理側キャンセル機能      | 講師が管理画面からキャンセル操作を行う                          | 📋   |
| `CancelReservation`共通化 | `cancelledBy`(client/teacher)を引数に取るアクションクラスへ集約 | 📋   |

### 1-4. マイページ

| 項目                   | 内容                                                                          | 状態 |
